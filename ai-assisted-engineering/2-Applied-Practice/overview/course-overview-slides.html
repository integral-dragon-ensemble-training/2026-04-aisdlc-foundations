OVERVIEW | COURSE THESIS
INTEGRAL DRAGON

AI-assisted engineering is most valuable when it helps teams see, reduce, and systematically pay down the friction and technical debt that make software hard to change.

The point is not more code. The point is easier change. A bad codebase plus AI does not become a good codebase — it becomes a faster bad codebase.

OVERVIEW | PROJECT HEALTH
INTEGRAL DRAGON

What good looks like — nine health categories

Category 01

Documentation & Discoverability

Docs are current enough to trust; new engineers find the path from clone to running system.

Category 02

Architecture Clarity

Boundaries, responsibilities, and dependency direction are legible at a glance.

Category 03

Reproducible Setup

Any clean machine reaches a working dev state with bounded effort.

Category 04

Testability & Confidence

Tests help the team move faster. Failing tests are actionable, not brittle.

Category 05

Build, Deploy & Release

Versions mean something. Deployed artifacts are identifiable and verifiable.

Category 06

Data & Schema Discipline

Schema changes are reviewable and repeatable. Test data is intentional.

Category 07

Observability

Logs, traces, and metrics answer what happened, where, when, and why.

Category 08

Security & Supply Chain

Secrets managed deliberately. Vulnerable dependencies are visible.

Category 09

Workflow & Collaboration

Small, reviewable changes land consistently. AI diffs are reviewed like any other.

OVERVIEW | DEBT TAXONOMY
INTEGRAL DRAGON

Technical debt, named in nine categories

Debt 01

Knowledge Debt

Outdated docs, no architecture map, unclear ownership, hidden operational steps.

Debt 02

Design Debt

Tangled dependencies, mixed responsibilities, duplicated logic, weak boundaries.

Debt 03

Environment Debt

Snowflake machines, missing setup automation, dependency version chaos.

Debt 04

Quality-Confidence Debt

Missing tests, flaky tests, meaningless coverage, no regression confidence.

Debt 05

Release Debt

Manual release steps, unknown deployed version, weak artifact traceability.

Debt 06

Data & Schema Debt

Untracked schema changes, fragile migrations, impossible test-data setup.

Debt 07

Observability Debt

Noisy logs, missing correlation, no traces, vanity dashboards.

Debt 08

Security & Supply Chain

Hardcoded secrets, unreviewed dependencies, no scanning, unclear provenance.

Debt 09

Workflow Debt

Huge PRs, weak review norms, cleanup never scheduled, AI changes under-reviewed.

OVERVIEW | AI USAGE DISCIPLINE
INTEGRAL DRAGON

Where AI helps — and where it is risky

AI IS STRONG WHEN

  • Summarizing a codebase or long scripts
  • Finding patterns, duplication, and inconsistencies
  • Identifying documentation and test gaps
  • Generating first-pass scaffolds and checklists
  • Proposing small, scoped refactors
  • Tracing cross-cutting logic and dependencies
  • Turning issue patterns into cleanup plans

AI IS RISKY WHEN

  • Rewriting large parts of a system without context
  • Writing tests that assert implementation trivia
  • Producing documentation that sounds right but is wrong
  • Making security-sensitive changes without review
  • Generating more code than the team can absorb
  • Normalizing insecure patterns seen in public repos
  • Hiding weak reasoning behind fluent summaries

Use AI to inspect, explain, propose, and scaffold. Use humans to decide, validate, and own the change. Treat AI output as draft material with a burden of proof.
