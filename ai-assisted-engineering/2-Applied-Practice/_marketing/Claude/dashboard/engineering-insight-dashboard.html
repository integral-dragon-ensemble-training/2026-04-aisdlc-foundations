Where the friction lives — top three hotspots

Observable friction, not opinions. Each item names who pays, how often, and what change is slowed.
Claims submission API · validation logic is entangled with persistence
Every new feature ~1.5× estimate · est. 18 engineer-hours / month
Claude found duplicated validation rules across SubmissionService, PayerMapper, and the controller layer. New feature work consistently pulls in changes to all three. Interest shows up as missed story-point commitments on the submission team.
Eligibility data pipeline · undocumented Oracle seed requirements
~6 hours per new engineer setup · est. 14 engineer-hours / month
Local setup passes but produces empty responses for 30% of eligibility codes. The required seed scripts live in an engineer's private Bitbucket fork. Interest shows up as new hires blocked for a day and a half before their first real ticket.
Denial-letter templates · release required for every legal copy change
3 legal changes / month · est. 10 engineer-hours / month
Templates are compiled into the binary. Legal routinely requests small copy edits. Each one becomes an engineering ticket, a release, and a deployment check. Interest shows up on both the engineering and compliance backlogs.

What Claude is watching next

Forward-looking insights the team should act on before they become fires.
01
Payer integration library is drifting from its consumer services
Three downstream services have forked the PayerAdapter interface over the last six weeks. Each fork represents hidden friction that will compound when the next payer onboards.
02
Denial-letter template system is a candidate for extraction
Legal requested three copy changes last month; each one triggered an engineering cycle. Moving templates out of the binary would remove this loop entirely.
03
The oldest two services have no trace correlation at all
When these services fail in production, the team debugs by intuition. If incident volume rises, the time-to-resolve on these two services will balloon first.
How to read this dashboard
Insights, not metrics.
We are not counting lines of AI-written code or completion-acceptance rates. Every tile on this dashboard reflects where engineering time is being spent, where confidence is weak, and what the team chose to improve.
Evidence from the codebase, judgment from the team.
Claude reads the repository, gathers evidence, and frames questions. Humans score, prioritize, and decide what to change. Every number on this page is defensible against the code it came from.
Designed to be re-runnable.
The same scorecard can be run monthly. Deltas are visible, not promised. If the team's investment in engineering health is paying off, this dashboard will show it — and if it isn't, it will show that too.
