WORKSHOP 03 | PROMPT SHAPE
INTEGRAL DRAGON

Weak prompt vs. tightened prompt

Before — weak

Generic ask, no structure, no criteria

Analyze this codebase and improve it.

Produces generic, plausible prose. No evidence trail. No ranked findings. Uncertainty hidden.

After — tightened

Scoped, criteria-driven, inspectable output

Act as a senior architect. Evaluate this Python/FastAPI repo against 9 project-health categories. For each: confirmable evidence, weaknesses, likely debt, 1–5 rating, confidence level. Rank the top 5 problems. Distinguish facts from inferences. Do not hallucinate architecture.

Produces structured findings with explicit uncertainty — an artifact a reviewer can check.

Five levers — scope · quality criteria · output structure · uncertainty · validation expectations.

AI for documentation and architecture discovery

One of the safest high-value uses of AI is making hidden knowledge explicit — then reviewing it against the repo.

Docs · 01

README refresh candidates

Generate draft sections grounded in the repo's actual structure. Flag what the current README claims but the code does not support.

Docs · 02

Module role summaries

One-paragraph summaries of each top-level module with its likely responsibilities and obvious collaborators.

Architecture · 03

C4-style component overview

First-pass container and component map. Treat as a draft needing human edit, not a source of truth.

Architecture · 04

Unanswered questions list

Explicit uncertainty — what the repo does not answer — so reviewers know where to look next.

INTEGRAL DRAGON

AI for tests and small refactors

AI accelerates confidence-building work when the scope is constrained. Small, scoped, reviewable — always.

  • →Candidate tests for one component · generate ideas, then filter for real behavior over implementation trivia.
  • →Seam creation · propose small isolation changes before attempting bigger refactors.
  • →Logging and error-handling improvements · repetitive cleanup under clear rules you can review line by line.
  • →Small, scoped diffs · split large proposals before merging anything — oversized diffs are a failure mode, not a feature.

WORKSHOP 03 | FAILURE MODES
INTEGRAL DRAGON

AI failure modes — the red flags

The biggest risk is fluent overreach. Fluency is not evidence. Participants need permission to distrust impressive-looking AI output.

Red flag 01

Hallucinated architecture

Confident descriptions of flows, services, or boundaries that don't exist in the repo.

Red flag 02

Shallow tests

Assertions that exercise lines without testing the behavior that actually matters.

Red flag 03

Unsafe refactors

Changes that silently expand scope beyond the stated problem.

Red flag 04

Oversized diffs

PRs too large for a reviewer to reason about in one sitting.

Red flag 05

Confident summaries

Smooth prose that hides uncertainty instead of surfacing it.

Red flag 06

Unrequested complexity

Generated abstractions, patterns, or frameworks no one asked for.

Framing line — Trust evidence, not fluency.

WORKSHOP 03 | HUMAN REVIEW RULES
INTEGRAL DRAGON

The review bar should be clearer, not lower

Rule 01

Require explicit scope

Name the files, the behavior, and the acceptance before the AI starts. Scope drift is the most common AI failure.

Rule 02

Require validation steps

Every AI change needs real tests, actual run output, or concrete verification. No evidence, no merge.

Rule 03

Prefer small diffs

Split large AI proposals into reviewable slices. If a human can't hold it in their head, it is not ready.

Rule 04

Check assumptions against the repo

Treat AI output as a draft, not as evidence. Verify every claim of "the repo does X" against the actual code.

This is where AI excitement becomes engineering discipline.
