# Five-Day Course Map

Each session is designed for 60 minutes. The daily flow intentionally repeats a pattern so students know what to expect.

## Standard daily structure

- 0–10 min: concept framing
- 10–25 min: examples and repo inspection
- 25–45 min: exercise or group activity
- 45–55 min: debrief and synthesis
- 55–60 min: reading / next-step setup

---

## Day 1 — What Good Looks Like

### Theme

Define software project health in practical terms.

### Learning objectives

Students should be able to:

- explain why “clean code” is too narrow,
- identify the major categories of project health,
- describe what good looks like in documentation, architecture, environment, quality, observability, and release discipline,
- use a lightweight scorecard to assess a real repository.

### Lesson blocks

1. Why AI-assisted engineering needs a quality target
2. Expanded framework for software project health
3. Repo health scorecard walkthrough
4. Group assessment exercise

### Suggested outcome

Students leave with a concrete rubric for judging a software project.

---

## Day 2 — Technical Debt as Friction

### Theme

Show that technical debt is broader than messy code and can be explained in terms of friction and change cost.

### Learning objectives

Students should be able to:

- define technical debt in operational terms,
- distinguish debt categories,
- explain “interest” using real engineering examples,
- prioritize debt based on impact on change, confidence, and flow.

### Lesson blocks

1. Technical debt taxonomy
2. Debt interest and cost of change
3. Friction mapping in a codebase or process
4. Prioritization exercise

### Suggested outcome

Students stop treating debt as a vague complaint and start treating it as a diagnosable cost structure.

---

## Day 3 — Using AI to Inspect and Improve

### Theme

AI is used to inspect, explain, plan, scaffold, and clean up targeted problems.

### Learning objectives

Students should be able to:

- identify high-value AI use cases in an existing codebase,
- write prompts that produce useful analysis instead of generic noise,
- use AI for documentation, codebase understanding, tests, and small refactors,
- recognize where AI is overreaching.

### Lesson blocks

1. Good AI-assisted engineering workflows
2. Prompt patterns for repo analysis
3. Small-scope refactor/test/doc examples
4. AI risk and review discussion

### Suggested outcome

Students see AI as a disciplined engineering tool, not a slot machine for random code generation.

---

## Day 4 — Validation: Tests, Observability, Security

### Theme

Improvement only counts if it is validated.

### Learning objectives

Students should be able to:

- explain why confidence beats raw speed,
- describe the role of tests, observability, release verification, and security checks,
- evaluate whether a project’s quality signals are trustworthy,
- propose baseline validation gates for AI-assisted changes.

### Lesson blocks

1. Testability and quality confidence
2. Observability as maintainability
3. Security and supply-chain hygiene
4. Validation checklist exercise

### Suggested outcome

Students learn that AI-generated improvements require stronger—not weaker—verification discipline.

---

## Day 5 — Making It Repeatable

### Theme

Turn one-off cleanup into normal engineering practice.

### Learning objectives

Students should be able to:

- design a staged improvement plan,
- choose what to fix first,
- create an ongoing repo health cadence,
- define review and workflow rules for AI-assisted engineering.

### Lesson blocks

1. Staged modernization instead of rewrite fantasies
2. What to standardize in workflow
3. Team scorecards, checklists, and debt backlogs
4. Final exercise: 30-day improvement plan

### Suggested outcome

Students leave with a repeatable practice they can apply to their real projects.
