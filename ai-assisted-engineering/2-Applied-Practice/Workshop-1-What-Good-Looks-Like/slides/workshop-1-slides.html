WORKSHOP 01 | MENTAL MODEL
INTEGRAL DRAGON

A healthy project has more than clean code

Clean code is one slice

Style, naming, and internal hygiene matter — but they are a small fraction of how changeable a system really is.

  • readable modules
  • reasonable names
  • low duplication

Project health is broader

Docs, environment setup, release process, and operational visibility are all part of software quality.

  • docs you can trust
  • setup that reproduces
  • runtime you can see
  • releases you can verify

Teams suffer when these are weak even if the code style looks decent.

Nine categories of project health

Category 01

Documentation & discoverability

A repo that tells the next engineer how to understand, run, and change it.

Category 02

Architecture clarity & boundaries

Clear responsibilities, dependency direction, useful diagrams.

Category 03

Reproducible developer setup

Clean-machine onboarding with bounded effort and no oral tradition.

Category 04

Testability & quality confidence

Tests that help you move, not vanity coverage. Confidence, not theater.

Category 05

Build, deploy & release discipline

Repeatable builds, clear versions, verifiable deployments.

Category 06

Data, schema & environment

Reviewable migrations, trustworthy seed data, reversible changes.

Category 07

Observability & operability

Logs, traces, metrics, and deploy visibility that support diagnosis.

Category 08

Security & supply chain

Secrets managed, dependencies scanned, secure defaults deliberate.

Category 09

Workflow & collaboration

Small reviewable changes, clear ownership, AI output reviewed like any diff.

WORKSHOP 01 | NUANCES
INTEGRAL DRAGON

Four nuances to teach in the first workshop

Testability

Not "run the full system locally"

For distributed systems, the better standard is meaningful local or isolated testability — not perfect local parity.

Confidence

Not "high coverage"

Coverage is a count. Confidence is a capability. Teach risk-relevant testing, not number-chasing.

Security

Not "no passwords in code"

Teach secrets management and secure default handling — including dependency and supply-chain hygiene.

Observability

Not "just logs"

Teach logs + traces + metrics + deploy visibility + debugging usefulness. It is part of maintainability.

WORKSHOP 01 | EXERCISE
INTEGRAL DRAGON

Exercise: Repository Health Scorecard

What you will do

  • Review a real repo: README, structure, workflows, visible docs.
  • Score each of the nine categories from 1 to 5.
  • Write one sentence explaining each low score.
  • Identify the three weaknesses most likely to slow future change.

How to use AI

  • Ask AI to summarize repo structure and surface missing docs.
  • Ask AI to list visible testing and CI signals.
  • Do not let AI assign scores — humans own the judgment.
  • Ground every score in actual repo evidence.

Output: a scored rubric, a top-three problem list, and a one-paragraph conclusion. Time: 15–20 minutes in pairs or small groups.
