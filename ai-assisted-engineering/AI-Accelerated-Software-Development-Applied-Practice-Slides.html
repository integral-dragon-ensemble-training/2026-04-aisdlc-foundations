What Applied Practice adds

From posture to output

Foundations instills the operating model. Applied Practice turns that model into concrete, reviewable artifacts inside real repositories.

From exercises to deliverables

Exercises disappear. Deliverables remain. Every week ends with an artifact that can be reviewed, inherited, and defended to engineering or security stakeholders.

From individual to team-scale

Cohort review, coaching, and artifact submission mirror the review loops of real engineering teams — not solo learning environments.

Workstream 01  |  Week 3
01
Codebase Analysis and Documentation
Week 3 ~4 hours Read-only first
You'll learn
→Safe repo inspection with read-only first-pass discipline
→Dependency and boundary mapping across system components
→Hotspot identification using churn, complexity, and low-test areas
→Documentation that is actionable for new engineers — not generic summary text
Inspect a production-style repository safely. Identify system boundaries. Create a risk-ranked backlog of safe improvements.
Deliverables
→Architecture README covering purpose, runtime model, risks, and key boundaries
→ADR v1 documenting one significant decision with trade-offs
→Dependency and boundary map notes
→Risk-ranked backlog with the top safe PR candidates
→One selected safe change with plan and acceptance criteria

Workstream 02  |  Week 4
02
Testing and Coverage Acceleration
Week 4 ~4 hours Unit tests · Deterministic patterns only
You'll learn
→Baseline coverage measurement and interpretation before adding tests
→High-ROI target selection based on risk and change frequency
→Human review loops for AI-generated tests — no blind trust
→Coverage delta reporting that communicates value clearly to stakeholders
Use AI where it improves confidence — not where it creates unmaintainable test debt. Readable tests survive future maintenance; generated noise does not.
Deliverables
→Coverage baseline before any changes are made
→Test plan defining what to test and why — not just what AI generated
→New tests with edge cases and readable, maintainable naming
→Coverage delta report after changes with clear before/after framing
→Refactoring notes where testability required code improvements

Security and Compliance
Week 5 ~4 hours Security review discipline · Compliance-ready evidence
You'll learn
→Prompt redaction and safe-data handling patterns for enterprise workflows

Observability
Week 6 ~4 hours Operational readiness · Reconstructable system behavior
You'll learn
→Structured logging with defined fields and stable, documented schemas

Applied Practice  |  Quality Standards
Assessment rubrics — what good looks like
PR Rubric
Clear statement of what changed
Clear explanation of why it changed
Risk assessment included
Testing approach documented
Follow-up items identified
Tests Rubric
Readable and well named — purpose is obvious
Deterministic and non-flaky
Edge cases included where appropriate
Maintainable structure that survives future changes
Meaningful confidence improvement shown
Documentation Rubric
Actionable for another engineer, not just the author
Accurate to current system state
Risks clearly identified — not hidden or minimized
Run and build steps verified where relevant
Trade-offs explained rather than concealed
Security Rubric
Sensitive-data boundaries are explicit
No unsafe prompt content normalized
Mitigation steps are concrete and reviewable
Approvals and ownership are visible
Evidence is sufficient for later audit
Observability rubric: Logs structured and purposeful · Correlation usable end-to-end · Runbook steps practical · Troubleshooting maps to real signals · Blind spots acknowledged explicitly
