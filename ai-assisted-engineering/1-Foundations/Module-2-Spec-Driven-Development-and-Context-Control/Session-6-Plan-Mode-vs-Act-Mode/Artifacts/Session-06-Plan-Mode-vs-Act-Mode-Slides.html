Session 06  |  Plan Mode
What Plan Mode is for

Read-only codebase analysis

Claude explores files, traces logic, and identifies boundaries without touching anything.

Clarifying question discovery

Missing requirements and hidden assumptions surface before a single line is written.

Multi-file change proposals

See the full proposed change surface before execution begins — with room to challenge it.

Safe exploration of sensitive areas

Auth, billing, compliance-sensitive paths — analyze the risk before committing to a direction.

Session 06  |  Trigger Conditions
Use Plan Mode when the task is high-risk or high-ambiguity
$ claude --permission-mode plan
→Multi-step implementation touching several files or systems
→Refactors that span multiple files or cross module boundaries
→Migrations, auth changes, billing logic, CI/CD, or compliance-sensitive flows
→Any work where the repo boundary is still unclear from the ticket alone

Session 06  |  Plan Quality
What a good plan should contain

Likely files and components touched

Explicit list — not "probably some auth files" but named paths and modules.

Invariants that must remain true

Schema stability, API contracts, test baseline, permissions model.

Migration or compatibility concerns

Any backward compatibility risk, rollout ordering, or flag dependency.

Testing strategy

Which tests will verify the change, what new tests are needed, and how to run them.

Open questions and assumptions

What the plan assumed without evidence — this is where reviewers should focus.

Session 06  |  Review Discipline
Review the plan like an engineer, not a spectator
→What did the plan assume without evidence? These are the highest-risk items.
→What failure modes did it ignore? Ask for explicit coverage of edge cases.
→Which files should not be touched? If they are not in the plan, confirm they are out of scope.
→What test evidence would prove this change is safe? Make it concrete before approving.

Approval is not a ceremony

A plan approval should be a defensible engineering judgment, not a rubber stamp. If you cannot explain why you approved it, the plan is not ready.

Session 06  |  Teaching Nuance
Claude Code has more than two permission modes — but one mental model governs them all
ModeBehaviorWhen to use
planRead-only analysis; no repo writesHigh-risk or high-ambiguity tasks
defaultPrompts for approval on writes and commandsNormal development work
acceptEditsAuto-approves file edits; prompts for commandsTrusted editing in known codebases
autoRuns autonomously within allowed tool setScripted or pipeline-driven runs
bypassPermissionsNo confirmations — full autonomyControlled automation only; never interactive
→Core mental model: analyze first, execute only after review. The mode determines how that boundary is enforced.

Session 06  |  Failure Modes
Common mistakes with Plan Mode

Approving a plan that never named the touched files

If you don't know what will change, you haven't reviewed the plan.

Letting the agent execute against ambiguity

An ambiguous request produces an ambiguous plan. Approve only when the acceptance criteria are clear.

Treating Plan Mode as ceremony

Running Plan Mode without reading the output doesn't add safety — it adds latency.

Using broad permissions to compensate for weak planning

bypassPermissions is not a substitute for understanding what the agent will do.
