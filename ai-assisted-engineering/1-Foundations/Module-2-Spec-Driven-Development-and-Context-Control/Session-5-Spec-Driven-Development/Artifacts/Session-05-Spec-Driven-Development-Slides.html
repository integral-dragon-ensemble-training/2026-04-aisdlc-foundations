Session 05  |  The Gap
Vibe coding optimizes for motion. SDD optimizes for convergence.

Vibe Coding

Prompt → inspect → retry → patch → hope. Motion is easy to produce. Correct output is not.

The cost of iteration without intent

Each retry loop burns review time, builds technical debt, and drifts further from the actual requirement.

Spec-Driven Development

Define target behavior, constraints, and checks before code exists. Then execute once.

The win

Not formality for its own sake. Fewer loops, more predictable output, reviewable diffs from the first attempt.

Session 05  |  Anatomy
A useful engineering spec is compact and executable

Goal

What outcome must exist when the work is done.

Constraints

What the agent must not violate — architecture, schema, performance, policy.

Interfaces

APIs, files, schemas, commands, or contracts touched or produced.

Acceptance Criteria

Binary checks that let a reviewer say pass or fail without asking the author.

Non-Goals

What is explicitly out of scope. Keeps the agent from widening the blast radius.

Session 05  |  Criteria Quality
Good acceptance criteria are observable
Weak
"make auth more robust"
Better
"expired bearer token returns 401 with the existing error envelope"
Weak
"improve performance"
Better
"no database schema changes; existing integration tests continue to pass"
→A criterion is good if a reviewer can check it without talking to the author.
→If it requires judgment to evaluate, it is still a goal, not a criterion.

Session 05  |  Spec Discipline
Write what must be true before guessing how it should be built
→Lead with behavior, boundaries, and risk.
→Add implementation constraints only when they are real requirements — not preferences.

Valid constraint

"Must use existing audit logger" — this is a real invariant the agent must respect.

Design advice, not a criterion

"Should probably use a service class" — valid opinion, wrong place for it.

Session 05  |  Template
Minimum viable spec template
1Problem statement — what is broken or missing
2Current boundary or affected area — what code is in scope
3Inputs and outputs — what the system receives and produces
4Constraints and invariants — what must remain true
5Edge cases and failure modes — known exceptions to handle
6Acceptance checks — binary pass/fail criteria
7Non-goals — explicit scope boundaries

Session 05  |  Before You Write
Repo reconnaissance comes before finalizing the spec
→Read the current code before freezing the request.
→Ask Claude Code to map the auth flow, touched files, and existing conventions.
→Then write the spec against the system that exists, not the system you imagine.

Why it matters

Specs written without reading the codebase encode imaginary constraints and miss real ones. Reconnaissance turns guesses into facts before any generation begins.

Session 05  |  Failure Modes
Common mistakes when writing specs

Treating a user story as a spec

User stories express intent. Specs express observable behavior and constraints. They are different artifacts.

Untestable criteria

Writing criteria that sound good but cannot be checked by a reviewer without interpretation.

Implementation smuggled in

Embedding design guesses in the acceptance section — now the agent is constrained by your preference, not the requirement.

Missing non-goals

Forgetting to scope out adjacent concerns lets the agent widen scope silently.
