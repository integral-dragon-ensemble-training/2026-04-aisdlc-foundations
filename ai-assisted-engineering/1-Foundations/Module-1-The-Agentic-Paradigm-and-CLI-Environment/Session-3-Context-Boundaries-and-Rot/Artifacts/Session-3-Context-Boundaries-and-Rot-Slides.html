Foundations · Session 03

Context Boundaries and Rot

Why more data does not automatically create better reasoning, and why bounded context is an engineering control rather than a prompt preference.

Signal Task-relevant files, rules, and facts
Noise Irrelevant files, retries, and old instructions
Pressure Latency, token cost, and weaker adherence
Outcome Fluent answers with lower grounding quality
Misconception

More data is not the same as better reasoning

  • Traditional software functions do not usually degrade from larger payloads.
  • LLMs do degrade when irrelevant context overwhelms the task signal.
  • Large context windows create possibility, not permission.

Just because a model can read 200k tokens does not mean it should.

Context rot starts when the signal-to-noise ratio drops below what the model can reliably prioritize.
Context Rot

What degrades when prompts get noisy

Latency

Time-to-first-token rises as the model processes more clutter.

Cost

Every irrelevant token still consumes budget.

Precision

The model starts blending unrelated concepts and APIs.

Symptom

Fluent answer, weak grounding, wrong dependency.

Cause

Too much low-value context competing for attention.

Repo Myth

The whole-repo dump is usually the wrong move

  • Fifty files create cross-pollination risk.
  • Irrelevant modules can leak their names and patterns into the answer.
  • Oversized prompts make it easier to hallucinate the wrong API surface.

Better pattern: send the minimum working set for the immediate task, not the maximum available context.

Hidden Context

The user prompt is only the tip of the iceberg

System Prompt

Persona, rules, output constraints, and formatting instructions.

Tool Schemas

JSON definitions that explain what each tool does and how to call it.

History

Earlier turns, retries, tool output, and session state.

Hidden context is necessary, but it means the cognitive bucket is already partially full before the user starts typing.
Budgeting

What hidden context changes operationally

Context source Why it exists What it costs
System prompt Keeps the agent aligned to behavior and rules Consumes tokens on every request
Tool schemas Teaches the model how to call tools safely Bloats context when too many tools are enabled
Transcript history Maintains continuity within a session Accumulates retries, noise, and old mistakes
State

Sessions are continuity, not memory

  • Resuming a session often means replaying the entire transcript.
  • This preserves continuity for the current task.
  • It also preserves obsolete branches, failures, and retries.
Pros

Easy continuity inside one active task.

Cons

Context rot compounds across long transcripts.

Memory

Explicit memory stores facts, not chat history

  • Useful for durable repo rules and architectural conventions.
  • Keeps future sessions cleaner than transcript replay.
  • Requires active judgment about what is worth remembering.

Best practice: use sessions for short continuity, use memory for repo-wide facts and long-lived rules.

NxM Problem

Monolithic agents accumulate hidden token debt

Mega-agent

Many tools, many repos, many schemas, many failure modes.

Specialized agent

One or two tools, narrow scope, cleaner prompt, better routing.

Architectural result

Routing becomes the control plane for reliability.

We stopped building monolithic applications as the default. The same logic now applies to agent design.
Patterns
Monolithic pattern

Broad reach, high context clutter, heavier hallucination risk, difficult to govern.

Distributed pattern

Narrow tool scope, cleaner prompts, stronger reliability, easier supervision.

Exercise

Context scoping challenge

  • Take the ticket: migrate user authentication to OAuth2.
  • Pick exactly 3 to 4 files for the working set.
  • Justify why `database/migrations` stays out.
  • Justify why `frontend/components` stays out.

The exercise is not about finding every possibly relevant file. It is about identifying the smallest context set that preserves task signal.

Takeaways

Key ideas to carry forward

  • Context quality matters more than context volume.
  • Hidden context must be treated as part of the token budget.
  • Sessions and memory are different state-management tools.
  • Specialized agents outperform mega-agents when boundaries are clear.
The best context is not the largest context. It is the smallest context that still preserves the task signal.
