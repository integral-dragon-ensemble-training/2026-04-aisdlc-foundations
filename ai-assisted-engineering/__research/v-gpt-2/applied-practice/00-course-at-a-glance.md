# Course at a Glance

## One-sentence thesis

Use Claude Code CLI to help engineers see, reduce, and systematically pay down technical debt so software becomes easier, safer, and cheaper to change.

## Five-day arc

| Day | Theme | Core question | Main artifact |
|---|---|---|---|
| 1 | What good looks like | How do we tell a healthy codebase from a fragile one? | Repo health scorecard |
| 2 | Architecture clarity + reproducibility | Can a new engineer understand, build, and run this system? | Architecture map + local setup gap list |
| 3 | Testability + data discipline | Can we make changes and know whether they worked? | Test improvement plan |
| 4 | Observability + release visibility | Can we see what the system is doing and what version is live? | Telemetry and deploy-verification plan |
| 5 | Safe change + operating model | How do we pay down debt continuously with Claude Code CLI instead of occasionally? | Incremental debt-paydown roadmap |

## Five teaching pillars

1. Architecture clarity
2. Reproducibility
3. Testability
4. Observability
5. Safe change

## Role of Claude Code CLI

Claude Code CLI is the working instrument for:
- codebase triage
- architecture mapping
- README and ADR drafting
- finding missing tests
- proposing incremental refactors
- generating or tightening scripts
- reviewing migrations and data setup
- surfacing observability gaps
- drafting rollout plans and PR descriptions

## What students should leave with

- a practical definition of software quality for brownfield systems
- a repeatable review checklist
- a set of Claude Code CLI prompt patterns
- a way to prioritize debt reduction instead of chasing broad rewrites
- a safer mental model for applying AI to real delivery work
